# ADR 006 — ORM: SQLModel + Alembic over raw sqlite3

**Status:** Accepted

## Context

The initial implementation used Python's built-in `sqlite3` directly. With Phase 1 adding several new fields to `holdings` and Phase 2 introducing multiple new tables (`options`, `option_rolls`, `closed_holdings`, `closed_options`, `watchlist`), the raw-SQL approach would require:

- Hand-written `CREATE TABLE`, `INSERT`, and `SELECT` statements per table
- Dict-based data passing with no type safety at the DB boundary
- Schema changes applied manually (running `ALTER TABLE` by hand, no version history)

## Decision

Migrate to **SQLModel** for the ORM layer and **Alembic** for schema migrations before any new features are added.

- Tables are declared as Python classes in `models.py` (`class Holding(SQLModel, table=True)`), giving typed attributes, IDE autocompletion, and Pydantic validation for free.
- `db.py` uses SQLModel `Session` context managers and typed queries (`select(Holding)`).
- `get_holdings()` returns `list[Holding]`; callers use attribute access (`.ticker`) not dict access.
- `app.py` constructs `Holding(...)` objects in the save block instead of raw dicts.
- Alembic tracks all schema changes as versioned migration scripts. `init_db()` calls `alembic upgrade head` on startup so the DB is always in sync with the code.

## Alternatives considered

**Peewee** — simpler API, good SQLite support. Rejected: SQLModel's Pydantic integration gives better type safety and is more ergonomic with the rest of the stack (FastAPI-style field declarations, `model_dump()`, etc.).

**SQLAlchemy (raw)** — SQLModel is built on SQLAlchemy 2.0. Using it directly skips SQLModel's field-declaration sugar and loses the typed dataclass behaviour.

**Keep raw `sqlite3` + add Alembic** — possible but leaves dict-based data passing throughout; type errors on new fields would only surface at runtime.

## Consequences

- Every schema change requires an Alembic migration script (`alembic revision --autogenerate -m "..."` + `alembic upgrade head`). Small discipline overhead, but prevents accidental data loss.
- Databases created before this migration must be stamped once (`alembic stamp head`) to mark them as already at the initial revision. Fresh installs run `alembic upgrade head` automatically via `init_db()`.
- SQLModel requires `__tablename__` to be set explicitly when the class name doesn't match the desired table name (e.g. `Holding` → `"holdings"`).
